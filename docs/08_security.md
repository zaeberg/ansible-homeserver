# Security

Документ описывает текущую модель безопасности homeserver: сетевой периметр, хостовые ограничения, защиту сервисов и работу с чувствительными значениями.

## Целевая модель

Сервер рассчитан на использование внутри домашней сети. Базовая схема безопасности строится на нескольких независимых слоях:

- отсутствие внешней публикации сервисов на роутере;
- host firewall с политикой `LAN-only`;
- доступ к web-интерфейсам через `Traefik`, а не через прямые host-порты;
- аутентификация для административных интерфейсов;
- минимизация привилегий для контейнеров и web UI там, где это возможно.

## Периметр

На сетевом уровне сервер не должен принимать произвольные подключения из интернета. Ожидаемая схема:

- на роутере нет port forwarding на homeserver;
- UPnP/NAT-PMP не используется для автопубликации портов;
- доступ к серверу осуществляется из локальной сети или через VPN;
- при наличии IPv6 входящий трафик также должен быть ограничен политикой роутера.

## UFW на хосте

`UFW` является основной защитой на уровне Ubuntu-хоста.

Политика по умолчанию:

- `incoming: deny`
- `outgoing: allow`
- `routed: deny`

Разрешённые порты:

- только из `192.168.1.0/24`:
  - `22/tcp`
  - `80/tcp`
  - `22000/tcp`
  - `22000/udp`
  - `21027/udp`

Практический смысл этой схемы:

- SSH доступен только из локальной сети;
- web-сервисы через `Traefik` доступны только из локальной сети;
- `Syncthing` не принимает sync/discovery трафик извне;
- даже если на роутере будет ошибочная публикация, сам хост остаётся закрытым.

## SSH и привилегии

Доступ к хосту организован через пользователя `admin`.

Текущие меры:

- парольная аутентификация по SSH отключена;
- вход `root` по SSH запрещён;
- используется только аутентификация по ключам;
- `admin` входит в группу `sudo`.

Текущее ограничение:

- `admin_nopasswd_sudo` пока включён.

Это удобно для первых этапов эксплуатации и Ansible-прогонов, но означает, что доступ к SSH-ключу `admin` практически эквивалентен root escalation. После ввода управляемого `sudo`-пароля через Vault рекомендуется переключить `admin_nopasswd_sudo` в `false`.

## Reverse proxy и web-интерфейсы

Все web-интерфейсы публикуются через `Traefik`.

Текущая схема:

- web UI сервисов не открываются на отдельных host-портах;
- `Traefik` dashboard не публикуется через прямой `:8080`;
- `Traefik` dashboard доступен только по `http://traefik.home.local/dashboard/`;
- для dashboard включён `Basic Auth`;
- `api.insecure` выключен.

Это даёт:

- единый входной контур;
- единое место для правил маршрутизации;
- возможность централизованно применять middleware;
- отсутствие отдельного обходного канала в виде прямого dashboard-порта.

## HTTP security headers

Через middleware `security-headers` для HTTP-сервисов включены базовые защитные заголовки:

- `X-Frame-Options`
- `X-Content-Type-Options`
- `Referrer-Policy`
- `X-XSS-Protection`

Middleware используется для:

- `Traefik` dashboard
- `Homepage`
- `Glances`
- `Syncthing`

HSTS принудительно не форсируется, потому что система пока работает по HTTP без включённого HTTPS.

## Syncthing

`Syncthing` работает в локальном режиме.

Текущая конфигурация:

- `globalAnnounceEnabled = false`
- `relaysEnabled = false`
- `natEnabled = false`
- `localAnnounceEnabled = true`
- `insecureAdminAccess = false`

Практический смысл:

- сервис не публикует себя во внешних discovery-сервисах;
- не использует relay-узлы;
- не пытается обходить сетевой периметр через NAT traversal;
- административный интерфейс доступен только через `Traefik`.

## Homepage и Docker API

`Homepage` по умолчанию не получает доступ к `docker.sock`.

Это важно, потому что даже read-only доступ к Docker API расширяет последствия компрометации web UI. Если docker-виджеты действительно потребуются, доступ должен включаться осознанно через `homepage_mount_docker_socket`.

`Glances` доступ к `docker.sock` сохраняет, потому что он нужен для контейнерного мониторинга.

## Секреты и локально сгенерированные пароли

Для некоторых сервисов система может сгенерировать пароль автоматически при первом деплое, если он не задан явно:

- `Traefik` dashboard
- `Syncthing` GUI

Файлы на хосте:

- `/opt/apps/traefik/dashboard-password.txt`
- `/opt/apps/syncthing/gui-password.txt`

Для воспроизводимой конфигурации и управляемой эксплуатации эти значения рекомендуется задавать через `Ansible Vault`, а не оставлять случайно сгенерированными навсегда.

Аналогичный подход применяется и к переменным `Homepage`: чувствительные значения не хранятся в роли как готовый `.env`, а должны поступать из Ansible vars, при необходимости через Vault.

## Что не закрыто полностью

Текущая схема существенно снижает поверхность атаки, но не является финальной.

Ограничения:

- HTTPS ещё не включён;
- `Let's Encrypt` каталог подготовлен, но ACME не используется;
- `admin_nopasswd_sudo` пока включён;
- модель безопасности предполагает, что внешний периметр также корректно ограничен на уровне роутера.

## Рекомендуемый следующий уровень

Следующие шаги для усиления системы:

- включить HTTPS в `Traefik`;
- вынести чувствительные значения в `Ansible Vault`;
- задать управляемый `admin_password_hash`;
- после проверки переключить `admin_nopasswd_sudo: false`;
- использовать VPN для удалённого доступа вместо прямой публикации сервисов.
